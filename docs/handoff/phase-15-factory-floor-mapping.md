## Context: what this feature is

The Grid-Sensor-Pipeline evolves from *telemetry-centric* to
*asset-centric*. A sensor event is no longer an abstract reading; it
resolves to a real (or simulated) factory-floor asset and its physical
context — building, floor, line, cell, zone, and indoor `(x, y)`.

```
Current:  sensor telemetry → anomaly/rule evaluation → alert
Target:   sensor telemetry → sensor-to-asset lookup → asset location lookup
          → zone/floor context enrichment → alert / NotifyOps response
```

Inspired by the ERIP emergency-response POC (GPS/GIS to locate and route
to resources), adapted indoors: GPS → per-floor local coordinates; road
graph → (future) floor path graph; the deterministic foundation is what
P15 builds.

---

## The one load-bearing rule (do not violate)

**Deterministic services own all factual mapping. The LLM never invents
locations.** Enrichment is pure, synchronous, in-memory lookups. No
Bedrock, no `@langchain/*`, no network call appears in the enrichment
path. The LLM layer (Phase 8/9) consumes the enriched `locationContext`
as structured input and only generates prose about it. If you find
yourself reaching for an LLM call inside `lib/factory-floor/`, stop —
that's the bug.

---

## CLAUDE.md invariants this feature must honor

- **#1 — validate at the I/O boundary.** Registry seed data and any
  external input are parsed through Zod (`schemas.ts`) before use.
- **#2 — no I/O in `lib/`.** The mapping logic in
  `src/lib/factory-floor/` has no AWS SDK calls. The registry loads
  bundled JSON seed files (filesystem read at module init is acceptable,
  same shape as static config; keep it out of the hot lookup path).
- **#3 — pure, independently testable logic.** `enrichTelemetryEvent()`
  is a pure function of (event, registry) → enriched event.
- **Hard rules:** no `any`, no `console.log` (use the Powertools
  `logger` singleton), no bare `catch`.

## P15.2 — Domain model + seed data

### Types — `src/lib/factory-floor/types.ts`

Mirror the JSDoc-heavy, zero-AWS style of `src/lib/types.ts`. Define:

**`Asset`**
- `assetId: string`
- `displayName: string`
- `assetType: string`
- `plantId: string`
- `buildingId: string`
- `floorId: string`
- `lineId: string`
- `cellId: string`
- `zoneId: string`
- `sensorIds: string[]`
- `connectedPlcId?: string`
- `safetyClassification: string`
- `maintenanceOwner: string`
- `location: { x: number; y: number; rotation: number; units: string }`

**`FloorMap`**
- `plantId: string`
- `buildingId: string`
- `floorId: string`
- `name: string`
- `units: string`
- `width: number`
- `height: number`
- `zones: Zone[]`

**`Zone`**
- `zoneId: string`
- `name: string`
- `zoneType: string`
- `polygon: Array<{ x: number; y: number }>`

**`SensorMapping`**
- `sensorId: string`
- `assetId: string`
- `metricType: string`
- `unit: string`
- `normalRange: { min: number; max: number }`
- `thresholdRange: { min: number; max: number }`

**`LocationContext`**
- `plantId, buildingId, floorId, lineId, cellId, zoneId: string`
- `x: number; y: number`

**`EnrichedTelemetryEvent`**
- `eventId: string`
- `sensorId: string`
- `assetId: string`
- `assetName: string`
- `severity: string`
- `metricName: string`
- `metricValue: number`
- `threshold: number`
- `timestamp: string` (ISO 8601)
- `locationContext: LocationContext | null`
- `enrichmentStatus: 'enriched' | 'sensor_unmapped' | 'asset_missing' | 'location_missing'`

> `enrichmentStatus` + nullable `locationContext` are how the
> missing-mapping fail-safe surfaces (decision log pre-flight 4). Keep
> them; tests assert on them.

### Schemas — `src/lib/factory-floor/schemas.ts`

Zod schemas for `Asset`, `FloorMap`, `Zone`, `SensorMapping`, and
`EnrichedTelemetryEvent`, with types **inferred** from the schemas (single
source of truth, same pattern as `validator.ts`). The registry loader
parses seed JSON through these so malformed seed data fails loudly at
load, not silently at lookup.

### Seed data — `data/factory-floor/`

New top-level `data/` directory (the repo has none today; synthetic
*telemetry* is code-generated by the simulator, distinct from static
reference data). Three files, internally consistent with the decision
log's worked example:

- `demo-floor-map.json` — at least `demo-plant-01 / building-a / floor-1`
  with `width`, `height`, `units: "meters"`, and a `zones[]` including
  `zone-line-1-cell-a` (a polygon containing point `(42.5, 18.2)`).
- `demo-assets.json` — includes `conveyor-02` (Conveyor 02) on
  `line-1 / cell-a`, `zoneId: zone-line-1-cell-a`,
  `location: { x: 42.5, y: 18.2, rotation: 0, units: "meters" }`,
  `sensorIds: ["temp-044", ...]`. Add a handful of other assets for a
  realistic floor.
- `demo-sensor-mappings.json` — includes
  `temp-044 → conveyor-02` (`metricType: "temperature"`, `unit: "degC"`,
  `normalRange`, `thresholdRange: { min: -Infinity-ish, max: 130 }`).
  Add mappings for the other seeded sensors. **Leave at least one asset
  sensor intentionally unmapped** so the fail-safe path has a fixture.

## P15.3 — Deterministic location enrichment service

### `src/lib/factory-floor/asset-registry.ts`

Loads + indexes the seed data once (lazy singleton, same pattern as the
`llm-client` lazy client). Exposes pure lookups:

- `findAssetBySensorId(sensorId): Asset | null` — via `SensorMapping`.
- `getAssetLocation(asset): { floorId, x, y, ... }` — from `Asset.location`.
- `getZoneForAsset(asset): Zone | null` — by `asset.zoneId` (and/or
  point-in-polygon against the floor's zones; `zoneId` lookup is enough
  for P15.3, polygon test is a documented extension).

No AWS SDK, no network. Index structures: `Map<sensorId, SensorMapping>`,
`Map<assetId, Asset>`, `Map<floorId, FloorMap>`.

### `src/lib/factory-floor/enrichment.ts`

```ts
enrichTelemetryEvent(event: BreachLikeEvent): EnrichedTelemetryEvent
```

Where the input carries at least `eventId, sensorId, metricName,
metricValue, threshold, severity, timestamp`. Behavior:

1. `findAssetBySensorId(event.sensorId)`. If null →
   `{ ...event, assetId: '', assetName: '', locationContext: null,
   enrichmentStatus: 'sensor_unmapped' }`. **No throw.**
2. Resolve `getAssetLocation(asset)` and `getZoneForAsset(asset)`.
3. Build `locationContext` from the asset's plant/building/floor/line/
   cell/zone + `(x, y)`.
4. Return the fully-populated `EnrichedTelemetryEvent` with
   `enrichmentStatus: 'enriched'`.

Pure function of (event, registry). The only logging is via the
Powertools `logger` singleton (structured, no `console.log`).

### Tests — `src/__tests__/asset-registry.test.ts` + `src/__tests__/factory-floor-enrichment.test.ts`

Jest, consistent with the repo. Cover all seven required cases:

1. `sensorId` maps to `assetId`.
2. `assetId` maps to its floor location `(x, y)`.
3. asset maps to its zone.
4. a full telemetry event enriches successfully (matches the worked
   example: `temp-044` → `conveyor-02` → `zone-line-1-cell-a`, `(42.5,
   18.2)`).
5. a **missing** sensor mapping is handled safely (returns
   `enrichmentStatus: 'sensor_unmapped'`, `locationContext: null`, **no
   throw**).
6. enrichment makes **zero LLM calls** — assert no `@langchain/*` /
   `llm-client` import is invoked (e.g., spy/mock `invokeStructured` and
   assert `not.toHaveBeenCalled()`, or assert the module graph has no LLM
   dependency).
7. the enriched `locationContext` is shaped as valid structured input for
   the NotifyOps/LangGraph layer (parses against the `EnrichedTelemetryEvent`
   Zod schema).

## Explicit non-goals (do NOT build)

- React / map UI for the factory floor.
- Indoor routing / path-graph logic.
- BLE / UWB worker-tracking integration.
- CAD / BIM file import.
- Any LLM-derived location inference.
- DynamoDB-backed registry (documented production migration; not now).
- A new docs organization that conflicts with existing conventions.

---

## Production migration notes (capture, don't build)

- **Registry storage.** JSON seed → DynamoDB `assets` / `floor-maps` /
  `sensor-mappings` tables when live edits matter. Swap the loader behind
  the `asset-registry` interface; lookups unchanged.
- **Zone resolution.** `zoneId` lookup → point-in-polygon against floor
  zones when assets move or zones overlap.
- **Pipeline wiring.** When ready, the enrichment service is invoked in
  the alert path (before the LangGraph handler) so `locationContext` is
  attached to the alert payload. P15.1–P15.3 deliver the library + tests;
  wiring it into the Step Functions / alert-handler flow is a later
  sub-phase.
